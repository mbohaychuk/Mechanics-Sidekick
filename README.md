# Mechanics Sidekick

A local command-line RAG assistant for mechanics. Upload PDF service manuals for a vehicle, open a job, and ask questions — all answers are grounded in your own documents and nothing leaves your machine.

Built with Python, SQLite, and Ollama. No API keys, no cloud dependency.

---

## How It Works

The workflow follows a simple hierarchy: **Vehicle → Documents → Job → Chat**

1. Add a vehicle to the database
2. Upload one or more PDF manuals — each is extracted page by page, split into overlapping text chunks, embedded via a local Ollama model, and stored in SQLite
3. Open a job against the vehicle (e.g. "Replace timing belt")
4. Ask questions in an interactive chat session — your question is embedded, the most relevant chunks are retrieved by cosine similarity, and the answer is generated by a local LLM with those chunks as context

At no point does any data leave your machine.

## Usage

Make sure Ollama is running in a separate terminal before ingesting documents or starting a chat:

```bash
ollama serve
```

### Add a vehicle

```bash
uv run mechanic-sidekick vehicle add
```

Prompts for year, make, model, engine, and an optional VIN and notes. Prints the assigned vehicle ID on success.

### Upload a manual

```bash
uv run mechanic-sidekick document add <vehicle_id> path/to/manual.pdf
```

Extracts text, chunks it, embeds each chunk, and stores everything in SQLite. The document is marked `ready` when complete or `failed` if something goes wrong. Large PDFs take a few minutes.

```bash
# Optional: label the document type
uv run mechanic-sidekick document add <vehicle_id> path/to/manual.pdf --type wiring_diagram
```

### Manage jobs

```bash
uv run mechanic-sidekick job add <vehicle_id>       # create a job
uv run mechanic-sidekick job list <vehicle_id>      # list jobs for a vehicle
uv run mechanic-sidekick job show <job_id>          # show job details
```

### Chat

```bash
# Interactive session (recommended)
uv run mechanic-sidekick chat start <job_id>

# Single question
uv run mechanic-sidekick chat ask <job_id> "What is the torque spec for the cylinder head bolts?"
```

The assistant cites which document and page each answer came from.

## Configuration

Defaults work out of the box. Override any setting via a `.env` file or environment variable.

| Variable | Default | Description |
|---|---|---|
| `CHAT_MODEL` | `gpt-oss:20b` | Ollama chat model |
| `EMBED_MODEL` | `qwen3-embedding:4b` | Ollama embedding model |
| `DB_PATH` | `./data/app.db` | SQLite database path |
| `DOCS_DIR` | `./data/documents` | PDF storage root |
| `CHUNK_SIZE` | `1000` | Words per chunk |
| `CHUNK_OVERLAP` | `200` | Overlap between consecutive chunks |
| `TOP_K_CHUNKS` | `5` | Chunks retrieved per question |
| `RECENT_MESSAGES` | `6` | Chat history window sent to the LLM |

## Design Decisions

### Contextual retrieval

The initial implementation embedded raw text chunks directly. When tested against a real vehicle with both 4.2L and 6.0L engine manuals loaded, the retriever consistently returned chunks from both variants for any engine-related question — the embeddings had no way to distinguish them because the chunk text itself was often identical ("tighten head bolts in sequence") regardless of engine size. The LLM would then answer about whichever variant happened to score highest or appeared first in the context.

The fix is contextual retrieval. Before embedding each chunk, the system makes a lightweight LLM call to generate a 1–2 sentence summary that situates the chunk in its document: which engine variant it applies to, what system it covers, and what specific information it contains. This context, along with a structured metadata header (`Document: filename | Page: N`), is prepended to the chunk text before the embedding is computed.

```
Document: 10-ENGINE ASSEMBLY 4.2L.pdf | Page: 45
This excerpt covers cylinder head bolt torque specifications and tightening
sequence for the 4.2L V8 engine during reassembly.

[original chunk text...]
```

The original chunk text is stored separately in the database and used for display in source citations. What gets embedded is the enriched version. As a result, embeddings for 4.2L and 6.0L content now point in meaningfully different directions in vector space, and retrieval precision improves without any changes to the retrieval or ranking logic.

The tradeoff is ingestion time: every chunk requires one LLM call during `document add`. For large document sets this adds minutes to the process, but it is a one-time cost per document.

## Known Limitations & Future Work

These are deliberate tradeoffs made to keep the initial implementation simple. Each one is a clear next step.

**Brute-force vector search**

At query time, all chunk embeddings for a vehicle are loaded from SQLite into memory and scored individually in Python. For a vehicle with many large manuals this can mean thousands of chunks and noticeable per-query latency (a few seconds). The fix is to replace this with a proper vector store — [ChromaDB](https://www.trychroma.com/) or [Qdrant](https://qdrant.tech/) both embed easily into a local setup and support approximate nearest-neighbor indexing, which would make retrieval effectively instantaneous regardless of corpus size.

**Embeddings stored as JSON strings**

Each chunk's embedding vector is stored as a JSON string in SQLite rather than a binary blob. This means every query pays a `json.loads` deserialization cost for every chunk in the candidate set. A straightforward improvement is to deserialize and cache the embeddings as a numpy matrix the first time the vehicle's chunks are loaded, avoiding redundant work across questions in the same session.

**No batch ingestion**

Adding 30 manuals means running `document add` 30 times. A `document add-batch <vehicle_id> ./manuals/` command that walks a directory and processes all PDFs in sequence (with per-file progress reporting) would be a straightforward addition.

**Retrieval is vehicle-scoped, not job-scoped**

All documents for a vehicle are searched on every question regardless of which job is open. For a vehicle with many different manuals this works fine, but for more targeted retrieval it would make sense to let users tag documents to specific job types or manually scope retrieval to a subset of documents.

**Fixed word-count chunking**

Text is split into fixed-size word-count windows with overlap. This is simple and predictable but ignores document structure — a chunk might cut across a procedure boundary, mixing the end of one step with the start of the next. Structure-aware chunking (splitting on headings, numbered steps, or tables detected via PDF font metadata) would produce more coherent chunks. A parent-child approach — embed small chunks for precise retrieval, send the surrounding larger chunk to the LLM for fuller context — would also improve answer quality for multi-row spec tables and multi-step procedures.

**No reranking**

The current pipeline embeds the question, retrieves the top K chunks by cosine similarity, and passes them directly to the LLM. A cross-encoder reranker (e.g. a small BERT-based model via `sentence-transformers`) applied after initial retrieval would meaningfully improve answer quality for ambiguous or multi-part questions, at the cost of a second model inference step.
